# OpenClaw 工作原则 v1.0

**创建时间**: 2026-04-02  
**适用范围**: 所有通过 ACP-Workflow 驱动的开发任务  
**核心目标**: 提高架构质量、编码质量、问题解决效率

---

## 📋 核心工作原则

### 1. 架构先行原则

**问题**: 重复返工、架构不一致、模块接口混乱

**原则**:
> 在开始编码前，必须先完成架构设计和接口定义。

**执行流程**:
```
1. 阅读现有代码 (理解上下文)
2. 绘制架构图 (模块划分、数据流、接口定义)
3. 制定实施计划 (任务分解、优先级、验收标准)
4. 架构评审 (自我评审或同行评审)
5. 开始编码
```

**检查清单**:
- [ ] 是否已阅读相关现有代码？
- [ ] 是否有架构图/模块划分图？
- [ ] 是否定义了清晰的模块接口？
- [ ] 是否制定了实施计划？
- [ ] 是否考虑了与现有系统的兼容性？

**交付物**:
- 架构设计文档 (Markdown + Mermaid 图)
- 接口定义文件 (头文件/接口描述)
- 实施计划 (任务分解 + 时间表)

---

### 2. 编译质量原则

**问题**: 编译错误重复出现、类型不匹配、IO 访问错误

**原则**:
> 每次提交前必须编译通过，零警告策略。

**执行流程**:
```
1. 编写代码
2. 立即编译 (增量编译)
3. 修复所有错误和警告
4. 运行单元测试
5. 提交代码
```

**检查清单**:
- [ ] 编译是否无错误？
- [ ] 编译是否无警告？
- [ ] 是否使用了正确的 IO 访问模式？
- [ ] 类型是否匹配？
- [ ] 字面量格式是否正确？

**常见错误预防**:
| 错误类型 | 预防措施 | 检查方法 |
|---------|---------|---------|
| IO 访问错误 | 始终使用 `io().port` | 代码审查 |
| 字面量错误 | 使用十进制 `_d` | 编译检查 |
| 类型不匹配 | 显式类型转换 | 编译检查 |
| 未定义宏 | 查阅框架文档 | 编译检查 |

---

### 3. 问题解决原则

**问题**: 问题解决效率低、重复错误、缺乏系统性

**原则**:
> 系统性分析问题，根因导向解决，避免表面修复。

**执行流程 (5 Whys)**:
```
1. 问题是什么？(现象描述)
2. 为什么会出现？(直接原因)
3. 为什么直接原因存在？(根本原因)
4. 如何防止再次发生？(系统性解决)
5. 如何验证解决？(验收标准)
```

**检查清单**:
- [ ] 是否已完整收集错误信息？
- [ ] 是否已分类错误类型？
- [ ] 是否已找到根本原因？
- [ ] 解决方案是否系统性？
- [ ] 是否有验证方法？

**错误分类模板**:
```markdown
## 错误分析

### 现象
- 错误信息：[完整错误输出]
- 触发条件：[什么操作导致错误]
- 影响范围：[哪些模块受影响]

### 根因
- 直接原因：[技术层面的直接原因]
- 根本原因：[设计/流程层面的原因]

### 解决方案
- 短期修复：[立即执行的修复]
- 长期预防：[防止再次发生的措施]

### 验证
- 验证方法：[如何验证修复有效]
- 验收标准：[什么情况下认为问题已解决]
```

### 4. 技能复用原则

**问题**: 重复造轮子、技能未沉淀、知识未固化

**原则**:
> 每次解决问题后，必须考虑是否可以创建/复用技能。

**执行流程**:
```
1. 问题是否已解决？
2. 是否有现成技能可用？
3. 是否可以创建新技能？
4. 技能是否已文档化？
5. 技能是否已分享？
```

**检查清单**:
- [ ] 是否搜索了现有技能？
- [ ] 问题是否具有通用性？
- [ ] 是否可以创建技能？
- [ ] 技能是否有完整文档？
- [ ] 技能是否已测试？

**技能创建决策树**:
```
问题是否已解决？
├─ 否 → 先解决问题
└─ 是 ↓
    是否有现成技能？
    ├─ 是 → 复用技能
    └─ 否 ↓
        问题是否通用？
        ├─ 否 → 记录到文档
        └─ 是 ↓
            创建技能
            ├── 编写 SKILL.md
            ├── 实现核心逻辑
            ├── 编写单元测试
            └── 发布到 ClawHub
```

**技能模板**:
```markdown

## 触发条件
[什么情况下使用此技能]

## 使用方法
[如何使用此技能]

## 实现细节
[技术实现细节]

## 示例
[使用示例]

## 测试
[测试方法]
```

### 5. 流程驱动原则

**问题**: 工作流程混乱、任务状态不清晰、协作效率低

**原则**:
> 所有复杂任务必须通过 ACP-Workflow 流程驱动，确保可追溯、可协作。

**执行流程**:
```
1. 任务分解 (拆解为子任务)
2. 创建 ACP 会话 (指定 Agent 和任务)
3. 监控进度 (定期检查状态)
4. 结果验收 (验证交付物)
5. 经验沉淀 (创建技能/文档)
```

**检查清单**:
- [ ] 任务是否已分解？
- [ ] ACP 会话是否已创建？
- [ ] 任务说明是否清晰？
- [ ] 验收标准是否明确？
- [ ] 是否有超时设置？

## 📊 质量度量指标

### 架构质量
| 指标 | 目标值 | 测量方法 |
|------|--------|---------|
| 模块耦合度 | 低 | 依赖图分析 |
| 接口清晰度 | 高 | 代码审查 |
| 文档完整性 | 100% | 文档检查 |

### 编码质量
| 指标 | 目标值 | 测量方法 |
|------|--------|---------|
| 编译通过率 | 100% | 编译检查 |
| 警告数量 | 0 | 编译输出 |
| 代码复用率 | >50% | 代码分析 |

### 问题解决效率
| 指标 | 目标值 | 测量方法 |
|------|--------|---------|
| 平均修复时间 | <2 小时 | 时间跟踪 |
| 重复错误率 | <10% | 错误统计 |
| 根因分析率 | 100% | 审查检查 |

### 技能沉淀
| 指标 | 目标值 | 测量方法 |
|------|--------|---------|
| 技能创建数 | ≥1/周 | 技能统计 |
| 技能复用率 | >30% | 使用统计 |
| 文档完整率 | 100% | 文档检查 |
