## 1. 双重检查锁定为什么会有问题

```cpp
Singleton* instance = nullptr;          // 1. 读取指针
std::mutex mtx;

Singleton* get() {
    if (instance == nullptr) {           // 2. 第一次检查（快速路径）
        std::lock_guard<std::mutex> lock(mtx);
        if (instance == nullptr) {      // 3. 第二次检查（加锁后）
            instance = new Singleton;   // 4. 构造函数可能尚未完成
        }
    }
    return instance;
}
```

* **指令重排**：在编译器/CPU 层面，`new Singleton` 可以被拆成三步  
  `① 分配内存 → ② 在那块内存上调用构造函数 → ③ 把指针写回 `instance`。  
  如果 ③ 在 ② 之前完成，另一个线程在第一次 `if (instance==nullptr)` 时会看到 **非空指针**，但对象仍未构造完毕（“野指针”）。

* **CPU 缓存**：即使编译器不重排，CPU 的 store‑buffer 也可能导致其他线程看到的 `instance` 已经非空，而对象的成员仍是默认值/未初始化。

只要没有 **显式的内存屏障**（如 `std::atomic` + `memory_order_acquire/release`），上面的代码 **在多线程下是不安全的**。  
因此我们需要 **更强的同步手段** 或 **语言层面的保证**。

### 2.1 代码（C++17）

```cpp
// singleton_meyer.hpp
#pragma once
#include <utility>

class Singleton {
public:
    // 禁止拷贝/赋值
    Singleton(const Singleton&) = delete;
    Singleton& operator=(const Singleton&) = delete;

    // 示例业务接口
    void doSomething() const;

    // 全局访问点（懒加载）
    static Singleton& getInstance() {
        // C++11 起，局部静态变量的初始化是线程安全的，
        // 编译器会插入“魔法”保护（guard variable），
        // 只会有一个线程执行初始化，其他线程会等待完成。
        static Singleton instance;   // ← 关键行
        return instance;
    }

private:
    // 只允许 getInstance() 创建对象
    Singleton() = default;               // 可根据需要在这里做复杂初始化
    ~Singleton() = default;

    // 示例成员
    int value_ = 42;
};

// 实现示例
inline void Singleton::doSomething() const {
    // 这里使用 value_ …
}
```

## 1️⃣ 数据竞争（Data Race）——所有队列操作都未加锁

```cpp
std::queue<int> sharedQueue;   // 1. 共享对象
void producer() {
    for (int i = 0; i < 100; i++) {
        if (sharedQueue.size() < 100) {      // 2. 读取
            sharedQueue.push(i);            // 3. 写入
        }
    }
}
void consumer() {
    while (true) {
        if (!sharedQueue.empty()) {          // 4. 读取
            int val = sharedQueue.front();   // 5. 读取
            sharedQueue.pop();               // 6. 写入
        }
    }
}
```

- **问题**：`push`、`pop`、`front`、`empty`、`size` 同时被多个线程（生产者、消费者）调用，而 C++ 标准库的 `std::queue` 不是线程安全的。  
- **后果**：同一时刻若有线程在写 `sharedQueue`，而另一线程在读或写，会产生 **数据竞争**（data race），导致程序行为不可预测（可能崩溃、读到损坏的数据、内存错误等），C++ 标准将此视为 **未定义行为**。

**Task 3: CUDA 调试**
当 CUDA kernel 启动返回 `cudaErrorInvalidConfiguration` 时，通常是因为启动配置参数（如 grid 大小、block 大小、共享内存大小等）不合法。以下是最常见的 5 个原因，按排查顺序排列：

---

### **1. Block 大小（threads per block）超出设备限制**
- **原因**：每个 block 的线程数超过设备允许的最大值（由 `cudaGetDeviceProperties` 的 `maxThreadsPerBlock` 返回）。  
- **常见场景**：例如在 RTX 3090（compute capability 8.6）上，`maxThreadsPerBlock = 1024`，如果设置 `threadsPerBlock = 1024` 且使用了 `dim3(32, 32)`，则会触发此错误（32×32=1024 合法，但如果设置为 `dim3(33, 33)` 则 33×33=1089 > 1024）。  
- **排查方法**：  
  ```cpp
  cudaDeviceProp prop;
  cudaGetDeviceProperties(&prop, deviceId);
  printf("Max threads per block: %d\n", prop.maxThreadsPerBlock);
  ```
  确认 block 大小不超过此限制。

---

### **2. Grid 大小或 block 大小未整除导致线程浪费（非致命但可能触发配置错误）**
- **原因**：如果 grid 大小设置过小，导致 `gridSize * blockSize < 所需总线程数`，可能不会直接报此错误，但若同时设置 `blockSize` 为不合理的值（如为 0 或超过硬件限制），则会触发。  
- **常见场景**：例如 `dim3 gridSize(0, 0, 1)` 或 `dim3 blockSize(0, 0, 0)`。  
- **排查方法**：  
  确认 grid 和 block 的每个维度 ≥ 1，且 block 大小不超过设备限制。

---

### **3. 动态共享内存大小设置非法**
- **原因**：在 `cudaLaunchKernel` 或 `<<<...>>>` 语法中，动态共享内存大小（第三个参数）设置为负数或超过设备可用共享内存上限（由 `cudaGetDeviceProperties` 的 `sharedMemPerBlock` 返回）。  
- **常见场景**：  
  ```cpp
  // 错误：分配过多共享内存
  size_t sharedMem = 100 * 1024; // 假设设备最大 48KB
  cudaLaunchKernel(kernel, dim3(1), dim3(256), sharedMem, stream);
  ```
- **排查方法**：  
  ```cpp
  printf("Max shared memory per block: %zu KB\n", prop.sharedMemPerBlock / 1024);
  ```
  确保分配值 ≤ 设备限制。

---

### **4. 使用的 CUDA 流（stream）不合法**
- **原因**：传入的 stream 未被正确创建或已被销毁。例如使用 `nullptr` 以外的无效指针。  
- **常见场景**：  
  ```cpp
  cudaStream_t stream = 0; // 使用默认流合法，但如果是自定义流，需确保有效
  cudaStream_t badStream;
  // badStream 未初始化或已销毁
  cudaLaunchKernel(..., badStream, ...); // 可能触发此错误
  ```
- **排查方法**：  
  确保所有 stream 通过 `cudaStreamCreate` 创建，且在销毁前使用。

---

### **5. Kernel 函数签名不匹配或设备函数（`__device__`）被错误调用**
- **原因**：如果尝试通过 `cudaLaunchKernel` 启动一个 `__device__` 函数（非 `__global__`），或参数类型/数量不匹配，会触发此错误。  
- **常见场景**：  
  ```cpp
  __device__ void myDeviceFunc(); // 仅限设备调用
  cudaLaunchKernel(myDeviceFunc, ...); // 错误：__device__ 函数不能从主机启动
  ```
- **排查方法**：  
  确认 kernel 定义为 `__global__`，且启动参数与函数签名完全匹配。

---

### **排查顺序建议**
1. **检查 block 大小**：确认 `threadsPerBlock` ≤ `maxThreadsPerBlock`。
2. **验证 grid/block 维度**：确保每个维度 ≥ 1，且 block 大小合法。
3. **检查共享内存大小**：确保 ≤ `sharedMemPerBlock`。
4. **审查 stream**：确认使用的 stream 有效。
5. **确认 kernel 函数定义**：确保是 `__global__` 且参数匹配。
