## 1️⃣ 原理概述

### 1.1 双重检查锁定的常见写法及其缺陷

```cpp
Singleton* getInstance() {
    if (p == nullptr) {               // 第 1 次检查
        std::lock_guard<std::mutex> lock(mtx);
        if (p == nullptr) {           // 第 2 次检查
            p = new Singleton;        // 分配 + 构造
        }
    }
    return p;
}
```

* **问题**：  
  - `new Singleton` 的语义是 **先分配内存，再构造对象，最后把指针写回 `p`**。编译器/CPU 可能把 **写指针** 放到 **对象构造完成之前**（指令重排）。  
  - 另一个线程在 `p != nullptr` 的第一次检查后拿到一个 **未完成构造的对象**，导致 **未定义行为**。  
* **解决思路**：必须保证 **指针的写入** 与 **对象的构造** 在所有线程看来是 **原子** 的，而且必须在 **所有线程** 看到同一份内存屏障后才能继续。

### 1.2 C++11/17 为局部静态对象提供的 “Magic Statics”

> **标准保证**（C++11 起的 6.7 [stmt.dcl] p4）  
> *If control enters the declaration concurrently while the variable is being initialized, the concurrent execution shall wait for completion of the initialization.*  

这意味着 **局部静态变量的初始化** 具备 **线程安全的“一次性”**（once‑and‑only‑once）语义，**不需要** 额外的 `mutex`/`atomic`/`call_once`：

* 编译器会在 **第一次调用**（控制流经过声明）时插入一个 **内存屏障**（通常使用 `std::call_once` 内部实现或平台提供的原子指令），保证 **初始化完全结束后** 才对其他线程可见。  
* 若初始化抛出异常，后续再次调用会 **重新尝试**（与 `std::call_once` 行为相同），因此不会出现 “永远不可用” 的情况。  

这就是 **Meyer’s Singleton**（Meyers 在 *Effective C++* 中提出的单例写法）能够“免锁、免双重检查”的根本原因。

### 1.3 `std::call_once` 的工作方式

`std::call_once` 使用 **一个 `std::once_flag` 对象** 与 **一个内部的锁**（通常是 `std::mutex`），保证：

1. **第一次调用** `call_once` 时会执行传入的 **Callable**（初始化代码），期间持有锁。  
2. **后续所有并发调用** 会被阻塞在 `call_once`，直到初始化完成，然后立即返回。  
3. 初始化过程同样受 **C++ 内存模型** 的保护——在 `call_once` 完成后，所有线程都会看到 **完整的初始化结果**（包括构造函数的副作用），不会出现 DCLP 的指令重排问题。  

因此，**使用 `std::call_once` 也可以安全地实现懒加载单例**，并且在 C++11 之前是主流做法。C++17 以后因为局部静态变量的线程安全保证，推荐直接使用 Meyer’s Singleton，除非你有 **更复杂的需求**（比如需要把单例实例存放在 **外部指针/引用** 中，或在 **非函数作用域** 中实现单例）。

## 1. **数据竞争（Data Race）——对 `sharedQueue` 的并发读写**

```cpp
std::queue<int> sharedQueue;   // 全局对象，多线程共享

void producer() {
    for (int i = 0; i < 100; i++) {
        if (sharedQueue.size() < 100) {
            sharedQueue.push(i);   // ❌ 同时有写操作
        }
    }
}

void consumer() {
    while (true) {
        if (!sharedQueue.empty()) {    // ❌ 同时有读操作
            int val = sharedQueue.front(); // ❌ 同时有读操作
            sharedQueue.pop();            // ❌ 同时有写操作
        }
    }
}
```

- **问题**：在 C++ 标准中，除非使用原子操作或互斥量等同步原语，**对同一对象（`sharedQueue`）的并发读写（至少有一个写）**构成 **数据竞争**，其行为是 **未定义**。在现代编译器和 CPU 上，这种竞争会导致：
  - 编译器可能对代码进行重排，导致业务逻辑错误。
  - CPU 缓存一致性协议被破坏，读取到 **脏数据**（过时的或部分更新的状态）。
  - 严重时会导致 **内存破坏**（比如 `queue` 内部的指针被同时修改），进而产生段错误或未捕获异常。

- **后果**：程序可能在某些平台上“看似”正常运行，但在另一次运行或不同硬件上崩溃。

## 2. **非原子性检查 + 更新（Check‑Then‑Act）导致的竞争条件**

```cpp
if (sharedQueue.size() < 100)   // ① 检查
{
    sharedQueue.push(i);        // ② 更新
}
```

- **问题**：① 与 ② 之间没有任何同步手段。线程 A 在检查 `size()` 为 99 后决定 `push`，而线程 B 同样在检查 `size()` 为 99 时也决定 `push`，两者会 **同时执行 `push`**，导致队列大小瞬间超过 100，甚至可能破坏内部链表的连接。

- **类似的竞争**：  
  ```cpp
  if (!sharedQueue.empty()) {    // ① 检查
      int val = sharedQueue.front(); // ② 读取
      sharedQueue.pop();            // ③ 删除
  }
  ```
  两个线程可能同时看到 `!empty()` 为 `true`，随后都执行 `front()`/`pop()`，导致 **读取到同一个元素**（数据重复）或 **丢失一个元素**（数据丢失）。

- **后果**：业务逻辑错误（生产者/消费者比例失控），甚至产生 **队列容量突破**、**双重释放**等更严重的内存错误。

---

## 3. ** Busy‑Wait（忙等） + 线程间缺乏通知机制**

- **问题**：消费者线程使用 **空轮询** 方式：
  ```cpp
  while (true) {
      if (!sharedQueue.empty()) {
          // ...
      }
  }
  ```
  这会 **无意义地占用 CPU**，并且在队列长时间为空时会导致 **“伪唤醒”**（即检查瞬间队列恰好为空而没有新的元素进入），线程会一直循环下去，**永远收不到真正的 “生产者已产生新数据” 的通知**。

- **后果**：CPU 资源被浪费；在实际系统中可能导致 **系统发热、功耗上升**，甚至在高并发场景下出现 **线程饥饿**（生产者没有机会获得 CPU）。

---

## 4. **缺少明确的结束条件——无限循环**

- **问题**：`consumer` 中的 `while (true)` 没有退出路径。即便 `sharedQueue` 在生产者结束后已经为空，**消费者仍会继续运行**，最终导致程序无法正常结束。

- **后果**：程序 **卡死**，需要强制终止或通过外部信号（如 `std::terminate`）终止线程。
