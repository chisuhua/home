# ADR 0001: 采用文档驱动开发流程

> **状态**: ACCEPTED  
> **日期**: 2026-03-18  
> **作者**: Sisyphus (Oh My OpenCode)  
> **适用范围**: 全局 (所有项目)

---

## 1. 决策内容

采用**文档驱动开发流程 (Document-Driven Development Flow)** 作为所有项目的标准开发方法。

## 2. 背景

随着项目复杂度增长，面临以下挑战：

- 需求理解不充分导致返工
- 架构决策未被记录，难以追溯
- 实现前缺乏充分讨论，设计缺陷发现晚
- 代码审查时机不一致

需要一个结构化流程来：
- 确保需求在实现前被充分讨论
- 记录架构决策及其原因
- 支持迭代和反馈
- 提高代码质量

## 3. 考虑的方案

### 方案 A: 文档驱动开发 (推荐)

**优点**:
- 需求清晰，减少返工
- 架构决策可追溯
- 早期发现问题
- 支持迭代和反馈
- 新人上手快 (文档齐全)

**缺点**:
- 前期投入时间较多
- 需要纪律性维护文档

### 方案 B: 轻量级流程

仅对大功能编写文档。

**优点**: 小功能快速迭代
**缺点**: 技术债务积累，文档不一致

### 方案 C: 无流程 (保持现状)

**优点**: 最大灵活性
**缺点**: 返工率高，新人上手慢

## 4. 决策结果

选择 **方案 A: 文档驱动开发**

**理由**:
- 复杂系统需要清晰架构
- 多 Agent 协作需要明确文档
- 前期投入可减少后期返工
- 符合 Superpowers 最佳实践

## 5. 权衡考虑

### 承认的权衡

1. **前期时间 vs 后期返工**
   - 接受前期多花 20-30% 时间
   - 预期减少 50%+ 返工

2. **文档完整性 vs 迭代速度**
   - 允许文档在迭代中更新
   - 但不允许无文档实现

3. **流程纪律 vs 灵活性**
   - 紧急 Bug 修复可事后补文档
   - 但需明确标记"TODO: 补文档"

## 6. 后果和影响

### 正面影响

- 需求理解更准确
- 架构决策可追溯 (ADR)
- 代码审查更高效
- 新人上手更快
- 技术债务可见化

### 负面影响

- 小功能开发周期变长 (前期)
- 需要维护文档一致性

### 技术债务

无新增技术债务。相反，该流程有助于**减少**技术债务。

## 7. 合规性

- ✅ 符合 Oh My OpenCode 规范
- ✅ 符合 Superpowers 技能体系
- ✅ 符合 C++ 行业最佳实践
