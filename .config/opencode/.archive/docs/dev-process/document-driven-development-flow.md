## 1. 概述

### 1.1 目标

本流程定义了一个**可迭代、文档驱动**的开发方法，确保：
- 需求在实现前被充分讨论和理解
- 架构决策被记录和追踪 (ADR)
- 计划在执行前经过审查
- 执行过程中可根据反馈调整需求和架构
- 所有变更经过验证和代码审查

### 1.2 适用场景

| 场景 | 流程适用度 | 说明 |
|------|-----------|------|
| 新功能开发 | ✅ 完整流程 | 从需求到上线 |
| Bug 修复 | ⚡ 简化流程 | 跳过头脑风暴，直接规划 |
| 架构重构 | ✅ 完整流程 + 额外审查 | 需要多轮 ADR |
| 小修改/配置 | ⚡ 最简流程 | 仅需计划和验证 |

### 1.3 文档产出物

```
docs/
├── superpowers/
│   ├── specs/           # 需求规格说明 (brainstorming 产出)
│   └── plans/           # 实现计划 (writing-plans 产出)
├── adr/                 # 架构决策记录 (ADR)
└── api/                 # API 文档 (Librarian 产出)

state/                   # 开发状态追踪
├── in_progress.lock
├── file_locks.json
├── progress.log
└── completed_{task_id}.json
```

## 2. 核心原则

### 2.1 文档优先 (Document First)

```
NO IMPLEMENTATION WITHOUT APPROVED DESIGN
```

- 设计文档未批准前，禁止编写实现代码
- 计划文档未审查前，禁止开始执行

### 2.2 可迭代 (Iterative)

流程不是线性的，允许回溯：

```
执行 → 发现问题 → 回溯到规划/架构/需求 → 调整 → 继续执行
```

### 2.3 TDD 强制 (Test-Driven)

- 新功能：必须先写测试
- Bug 修复：必须先写复现测试
- 重构：必须先有覆盖测试

### 2.4 频繁审查 (Review Often)

| 审查点 | 审查者 | 产出 |
|--------|--------|------|
| 设计文档 | 用户 + Spec Reviewer | 批准的设计文档 |
| 实现计划 | Plan Reviewer | 批准的计划 |
| 每任务完成 | Code Reviewer | 审查报告 |
| 功能完成 | Code Reviewer + 用户 | 合并决定 |

## 4. 阶段一：需求讨论与头脑风暴

### 4.1 触发条件

- 用户提出新功能需求
- 涉及 ≥3 个文件的修改

### 4.2 执行流程

1. **探索项目上下文** - 读取关键文件
2. **提供视觉助手** (如需要) - 单独消息
3. **逐一澄清问题** - 一次一个，优先选择题
4. **提出 2-3 种方案** - 含权衡分析
5. **呈现设计并批准** - 逐节确认
6. **编写设计文档** - `docs/superpowers/specs/`
7. **Spec 审查循环** - 最多 3 轮
8. **用户批准** - 进入阶段二

### 4.3 技能
- `superpowers/brainstorming`

---

## 5. 阶段二：架构设计与 ADR

### 5.1 何时编写 ADR

**必须**: 核心技术栈、模块边界、数据格式、安全决策

**不需要**: 变量命名、代码风格

### 5.2 执行流程

1. **识别决策点** - 在设计文档中标记
2. **编写 ADR** - `docs/adr/NNNN-<title>.md`
3. **ADR 审查** - Prometheus + 用户
4. **生成架构图** - Mermaid
5. **更新架构文档**

### 5.3 ADR 状态

```
PROPOSED → ACCEPTED → SUPERSEDED
```

## 6. 阶段三：详细规划

### 6.1 文件结构规划

- 每个文件一个职责
- 相关文件放在一起
- 遵循现有模式

### 6.2 任务分解

**颗粒度**: 2-5 分钟/步，一个动作

```markdown
### Task 1: [组件]
- [ ] Step 1: 写测试
- [ ] Step 2: 运行确认失败
- [ ] Step 3: 写实现
- [ ] Step 4: 运行确认通过
- [ ] Step 5: 提交
```

### 6.3 计划审查

- 派遣 plan-document-reviewer
- 最多 3 轮审查
- 选择执行模式 (子代理/内联)

---

## 7. 阶段四：执行与迭代

### 7.1 执行模式

**子代理驱动 (推荐)**:
- 每任务 fresh subagent
- 隔离上下文
- 并行执行

**内联执行**:
- 当前会话执行
- 适合小任务

### 7.2 TDD 循环

```
RED → GREEN → REFACTOR → Repeat
```

### 7.3 代码审查

**时机**: 每任务完成后

**等级**:
- Critical → 立即修复
- Important → 继续前修复
- Minor → 记录待处理

### 7.4 反馈调整

| 情况 | 回溯到 |
|------|--------|
| 新需求 | 阶段一 |
| 架构问题 | 阶段二 |
| 计划问题 | 阶段三 |

## 8. 阶段五：验证与完成

### 8.1 测试验证

```bash
cd build && ctest
```

### 8.2 完成选项

```
1. Merge locally     - 本地合并
2. Create PR         - 创建 PR
3. Keep as-is        - 保留分支
4. Discard           - 丢弃 (需确认)
```

## 10. 多项目使用指南

### 10.1 全局配置位置

```
~/.config/opencode/docs/
├── dev-process/
│   ├── README.md
│   ├── quick-reference.md
│   └── document-driven-development-flow.md
└── adr/
    └── 0001-document-driven-development.md
```

### 10.2 项目级引用

在项目 README 中添加:
```markdown
## 开发流程

本项目遵循 [文档驱动开发流程](~/.config/opencode/docs/dev-process/README.md)。
```

### 10.3 项目级覆盖

如项目有特殊需求:
```
<ProjectRoot>/docs/dev-process/local-overrides.md
```

### 10.4 跨项目 ADR

全局 ADR 适用于所有项目，项目特定 ADR 放在:
```
<ProjectRoot>/docs/adr/
```
