## Critical Learnings (Architecture Patterns)

### IO Port Usage Patterns

**WRONG (causes compilation errors)**:
```cpp
// Cannot assign directly in testbench
top.io().push_valid = ch_bool(true);

// Cannot pass IO ports to functions expecting ch_bool
auto result = sync_fifo(io().push_valid, ...);

// Cannot use && with IO ports
ch_bool will_write = wren && !is_full(count);  // Fails
```

**CORRECT (official pattern from samples/)**:
```cpp
// In testbench - use Simulator API
sim.set_input_value(top_device.instance().io().push, true);
auto val = sim.get_port_value(top_device.instance().io().dout);

// In describe() - IO ports work with select()
auto writing = io().push;  // No conversion needed
rd_ptr->next = select(writing, rd_ptr + 1_b, rd_ptr);

// For && operations, use select()
ch_bool will_write = select(wren, ch_bool(!is_full(count)), ch_bool(false));
```

### Module Instantiation

**CORRECT pattern**:
```cpp
// Use ch::ch_module for submodule instantiation
ch::ch_module<FiFo<ch_uint<8>, 16>> fifo_inst{"fifo_inst"};
fifo_inst.io().din <<= io().din;
io().dout <<= fifo_inst.io().dout;
```

### Simulation Semantics

- `tick()` advances simulation, then read values
- Cycle 0 = after first tick, values reflect initial state changes
- Counter example: Cycle N shows value N+1 (counter increments on tick)

### Shift Register Pattern (for Width Adapter)

```cpp
// Multi-byte shift register
ch_reg<ch_uint<8>> byte0(0_d);
ch_reg<ch_uint<8>> byte1(0_d);

byte0->next = select(shift_en, io().din, byte0);
byte1->next = select(shift_en, byte0, byte1);

// Concatenate for wider output
auto tmp = concat(byte1, byte0);  // 16-bit result
```

## Architecture Defects Found & Resolved

### Issue #1: IO Port Type Conversion
**Status**: ✅ Resolved (usage pattern documented)
**Problem**: `ch_in<ch_bool>` doesn't convert to `ch_bool`
**Solution**: Use `select()` instead of `&&`, or use Simulator API in testbench

### Issue #2: sync_fifo Write Enable Logic  
**Status**: ⚠️ Workaround applied
**Problem**: `write_enable = wren && !is_full(count)` uses old count value
**Impact**: FIFO may not write correctly in edge cases
**Workaround**: Use simplified write logic, external backpressure handling

### Issue #3: FIFO Empty/Full Detection
**Status**: 🟡 Known bug in official implementation
**Problem**: After draining, shows `empty=0, full=1` (incorrect)
**Impact**: Visual only, doesn't affect functionality
**Root cause**: Pointer comparison logic in `samples/fifo_example.cpp`
